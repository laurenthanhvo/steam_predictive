Introduction

Main predictive task

Problem: Given a usergame pair with its review and playtime information,
will the user recommend this game?

The input features for our main model include:

  • Total hours played (playtime_forever)
  • Recent hours played (playtime_2weeks)
  • Processed review text (TFIDF representation)
  • Basic user history (e.g., how many games the user owns, how many reviews they write)

The output of our model is a binary label:

  • 1 = user recommends the game
  • 0 = user does not recommend the game

This is a standard supervised learning classification task and aligns directly with the models covered in the course.

Bonus exploratory task: nextgame prediction

In a later section, we also experiment with a harder recommendation problem:

Given a user's past sequence of games, what is the next game they are likely to buy?

For this task, the output is a multi-class label over candidate games, and we evaluate accuracy, F1 score, and top-5 accuracy. This section is exploratory and illustrates the increased difficulty of full recommendation/ranking compared to the simpler binary recommend or not prediction.

What this notebook does

This notebook builds and evaluates a Steam game recommender model using the Steam AU Reviews and Steam AU Items datasets.

Concretely, it:

  • Loads and preprocesses the data

    • Expands the nested reviews and items lists into long-format tables (reviews_long and items_long).
    • Merges them into a single usergame interactions table containing: review text, the Steam Recommended flag, and playtime statistics.
    • Cleans and normalizes fields such as timestamps, numeric playtime, and the binary recommend label.

  • Performs exploratory data analysis (EDA)

    • Summarizes how many games users own and review.
    • Examines the distribution of total playtime, review length, and the proportion of recommended vs. non-recommended games.
    • Visualizes relationships such as playtime vs. recommendation and review length vs. recommendation.

  • Trains and compares predictive models for the main task

    • Main task: predict whether a user will recommend a game (binary label 1/0).
    • Builds TF-IDF features from review text plus numeric features such as playtime.
    • Trains several models:
      • A stratified random baseline (majority/coin-flip)
      • Logistic Regression + TF-IDF
      • Multinomial Naive Bayes + TF-IDF
      • LinearSVC + TF-IDF + numeric features (final model)
    • Evaluates models using accuracy and F1 score on a held-out test set and summarizes results in tables and plots.

  • Explores a harder next-game prediction task (bonus section)

    • Reformulates the problem as predicting which game a user will buy next among many candidates.
    • Constructs training pairs and evaluates several classifiers (Dummy, Naive Bayes, Logistic Regression, Linear SVC) using accuracy, F1, and top-5 accuracy.
    • Shows that performance on this multi-class next-game task is more limited, highlighting the challenges of full recommendation/ranking compared to the simpler binary recommend or not prediction.

DATA_DIR = Path('data')

def load_python_dicts_gz(path: Path, max_rows=None, verbose=True) -> pd.DataFrame:
    rows = []
    with gzip.open(path, 'rt', encoding='utf-8') as f:
        for i, line in enumerate(f, start=1):
            line = line.strip()
            if not line:
                continue
            rows.append(ast.literal_eval(line))
            if max_rows is not None and len(rows) >= max_rows:
                break
            if verbose and i % 100_000 == 0:
                print(f"Read {i} lines from {path.name}...")

    df = pd.json_normalize(rows)
    return df

Evaluation

Recall that our goal is to correctly predict whether a Steam user will recommend a game based on their review and gameplay behavior. Since this is a binary classification task, our evaluation needs to measure how well our model distinguishes between positive and negative recommendations.

As mentioned in the beginning of the notebook, our plans were to use the following metrics to evaluate our models:

  • Accuracy

    • Measuring accuracy will allow us to see the percentage of correct predictions our model makes, which is useful when classes (recommend vs. not recommend) are fairly balanced.

  • F1 Score

    • The F1 score represents the harmonic mean for precision and recall. This value is more robust than accuracy if our dataset is imbalanced, which can apply to Steam reviews.

We chose these metrics in particular due to the following reasons:

  • Recommendation data often contains more positive reviews than negative ones
  • Accuracy alone, especially at a high value, may be misleading for models that ignore the minority class
  • F1 score captures how well the model handles both sides, making it the most appropriate metric for this type of task

    records = []

In order to train a model that can predict the next game a user will review, we need to convert each user's review history into ordered input-label pairs.

In this step, we will:

  • Group reviews by user_id to keep each player's timeline intact
  • Sort each user's reviews by timestamp (posted_dt)
  • For users with 2+ reviews, generate sequential pairs:
    • Input: review_text of game i
    • Label: item_id of game i+1
  • A sequence may look like the following:
    • review of Game A next game B
    • review of Game B next game C
  • The function produces three aligned Series:
    • X_text: review text inputs
    • X_curr: current game ID (optional feature)
    • y_next: next game label the model must predict
In[57]:
def build_pairs(df):
    X_text, X_curr, y_next = [], [], []

    for uid, g in df.groupby("user_id"):
        g = g.sort_values("posted_dt")

        items = g["item_id"].tolist()
        texts = g["review_text"].tolist()

        if len(items) < 2:
            continue

        for i in range(len(items) - 1):
            X_text.append(texts[i])
            X_curr.append(items[i])
            y_next.append(items[i + 1])

    return pd.Series(X_text), pd.Series(X_curr), pd.Series(y_next)

Feature Engineering

Next, we will turn our review text into usable machine learning features by applying TF-IDF, a standard preprocessing technique.

Our function build_tfidf() will do the following:

  • Transform raw text into numeric vectors that capture the importance of each word in a review
  • Remove English stopwords (i.e "the" or "and) to reduce noise
  • Limit vocabulary size (max_features = 30000) to keep training fast and memory-efficient
  • Fit TF-IDF only on the training split to avoid data leaks, then transform the test split using the same vocabulary

As a result, this produces clean, high-signal text features that allow our baseline models (Naive Bayes, logistic regression, SVM) to perform well in our prediction task.

def build_tfidf(Xtrain, Xtest):

    tfidf = TfidfVectorizer(

    )

def top5(model, Xte, ytrue):
    if not hasattr(model, "predict_proba"):
        return None
    probs = model.predict_proba(Xte)
    idx = np.argsort(probs, axis=1)[:, -5:]
    labels = model.classes_[idx]
    return np.mean([ytrue[i] in labels[i] for i in range(len(ytrue))])


def evaluate(model, name, Xte, ytest):
    pred = model.predict(Xte)

    acc = accuracy_score(ytest, pred)
    f1 = f1_score(ytest, pred, average="weighted")
    t5 = top5(model, Xte, ytest)

    print(f"\n{name}")
    print("accuracy:", round(acc, 4))
    print("f1:", round(f1, 4))
    print("top5:", t5)

    return acc, f1, t5

Limitations

Despite using review text and playtime information, the overall accuracy across all models remains low. This reflects the fundamental difficulty of predicting a users next gameplayers often jump across genres, try new releases, or play games based on social influence or sales, all of which introduce randomness that the model cannot observe.

While playtime is included in the dataset, in this setup it is not directly integrated into the predictive features, so the models cannot actually learn from it. Even if it were added, raw playtime alone does not fully capture user preferencesome players try games briefly, leave early reviews, or accumulate hours over years, making playtime noisy and inconsistent.

Another major limitation is the extreme class imbalance. A few games appear very frequently, while most titles appear only a handful of times. This causes the baseline predictor to look deceptively strong and makes it difficult for models to learn meaningful patterns for less common games.

Text-based TF-IDF features also limit the understanding of game similarity. The models treat each review independently and cannot detect deeper patterns in genre, mechanics, or themes. Linear SVC, while strong, cannot output probabilities, which is why its Top-5 score is missing.

Areas for Improvement

To meaningfully boost accuracy, the model would benefit from richer, more structured information, including:

  • Using playtime more effectively
    • Incorporating it as a feature or engineer signals like long-term engagement, short trial, or review-to-playtime ratio.
  • Adding game metadata
    • Genre, tags, popularity, release year, ESRB rating, or platform would help the model recognize similarity between games.
  • Using user history
    • Instead of only one review next game, use sequences (RNN/Transformer-based recommenders like SASRec or GRU4Rec).
  • Better text representations
    • Replace TF-IDF with BERT/SentenceTransformer embeddings trained on gaming content.
  • Handling imbalance
    • Class weighting, oversampling, or grouping games into categories to stabilize predictions.

Conclusion

Our main predictive task was to determine whether a user would recommend a given game based on review text and basic playtime/user features. Using TFIDF representations of reviews together with numeric features such as total playtime, we found that simple linear models already perform very well: our best model, a LinearSVC with TFIDF and numeric features, achieves around 9192% accuracy and a F1 score of about 0.96 on a held-out test set, clearly outperforming a stratified random baseline and a Naive Bayes classifier.

The exploratory data analysis suggests why these models work: recommended games tend to have much higher total playtime, and non-recommended games are often accompanied by longer, more detailed negative reviews. These patterns provide strong, informative signals that a linear classifier can exploit.

In a bonus section, we also attempted a much harder next-game prediction task: given a users past sequence of games, predict the next game they will buy. Despite using similar features, performance on this multi-class setting is much weaker, reflecting the high variance and sparsity of this problem. Making next-game prediction reliable would likely require richer user profiles, game-level metadata (e.g., price, genre, release date, popularity), and sequence-aware models tailored to recommendation.

Overall, the notebook shows that text-plus-playtime features are sufficient to build a strong binary recommendation model, while also highlighting the limitations of our current setup for full next-game recommendation.
